개발 환경 구축: 개발 프로젝트를 이해하고 소프트웨어 및 하드웨어 장비를 구축하는 것. 

하드웨어 환경: 사용자와의 인터페이스 역할을 하는 클라이언트 그리고 클라이언트와 통신하여 서비스를 제공하는 서버로 구성됨. 
클라이언트: PC, 스마트폰
웹서버(Web Server): 클라이언트로 부터 직접 요청을 받아 처리하는 서버, 저용량의 정적 파일(HTML, CSS 등)들을 제공함.
웹 어플리케이션 서버(WAS, Web Application Server): 사용자에게 동적 서비스(사용자의 입력에 따라 다른 결과를 보여주는 서비스)를 제공하기 위해 웹 서버로부터 요청을 받아 데이터 가공 작업을 수행하거나, 웹 서버와 데이터베이스 서버 또는 웹 서버와 파일 서버 사이에서 인터페이스 역할을 수행
데이터베이스 서버(DB Server): 데이터베이스와 이를 관리하는 DBMS를 운영하는 서버
파일 서버(File Server): 데이터베이스에 저장하기에는 비효율적이거나 서비스 제공을 목적으로 유지하는 파일들을 저장하는 서버

소프트웨어 환경: 클라이언트와 서버 운영을 위한 시스템 소프트웨어와 개발에 사용되는 개발 소프트웨어로 구성. 
개발 소프트웨어 종류
요구사항 관리 도구: 요구사항의 수집과 분석, 추적 등을 편하게 도와주는 소프트웨어
설계/모델링 도구: UML(통합 모델링 언어)을 지원하며, 개발의 전 과정에서 설계 및 모델링을 도와주는 소프트웨어
구현 도구:개발 언어를 통해 애플리케이션의 실제 구현을 지원하는 소프트웨어
빌드 도구: 구현 도구를 통해 작성된 소스의 빌드 및 배포, 라이브러리 관리를 지원하는 소프트웨어
테스트 도구: 모듈들이 요구사하에 적합하게 구현되었는지 테스트하는 소프트웨어
형상 관리 도구: 산출물을 버전별로 관리하여 품질 향상을 지원하는 소프트웨어

개발 언어 선정 기준(적이 친범효)
적절성: 개발하려는 소프트웨어의 목적에 적합해야 함.
이식성: 다양한 시스템 및 환경에 적용이 가능해야 함.
친밀성: 개발 언어에 대한 개발자들의 이해도와 활용도가 높아야 함. 
범용성: 다른 개발 사례가 존재하고 여러 분야에서 활용되고 있어야 함. 
효율성: 코드의 작성 및 구현이 효율적이어야 함. 

소프트웨어 아키텍처
소프트웨어를 구성하는 요소들간의 관계를 표현하는 시스템의 구조

모듈화(Modularity): 시스템의 기능을 모듈단위로 나누는 것

추상화(Abstraction): 전체적이고 포괄적인 개념을 설계한 후 차례로 세분화하여 구체화시켜 나가는 것
과정 추상화
데이터 추상화
제어 추상화

단계적 분해(Stepwise Refinement): 상위의 중요개념부터 하위의 개념으로 구체화 시키는 분할 기법
정보 은닉(Information Hiding): 모듈내부에 정보가 감춰져 다른 모듈이 접근하지 못하도록 하는 기법

상위 설계
아키텍처 설계, 예비 설계
(구조, db, 인터페이스)

하위 설계
모듈 설계, 상세 설계
(컴포넌트, 자료구조 ,알고리즘)

협약에 의한 설계는 컴포넌트를 설계할 때 클래스에 대한 여러 가정을 공유할 수 있도록 명세한 것
선행조건(Precondition) 
결과 조건(Postcondition)
불변 조건(Invariant)

아키텍처 패턴; 아키텍처 설계할 때 참조할 수 있는 전형적인 해결 방식
레이어 패턴(Layers Pattern): 시스템을 계층으로 구분하여 구성하는 패턴
클라이언트 서버 패턴(Client-Server Pattern): 하나의 서버컴포넌트와 다수의 클라이언트 컴포넌트로 구성되는 패턴
파이프 필터 패턴(Pipe-Filter Pattern): 데이터 스트림 절차를 필터로 캡슐화 해 파이프를 통해 전송하는 패턴
모델 뷰 컨트롤러 패턴(Model-View-Controleler Pattern): 서브시스템을 모델, 뷰 ,컨틀롤러로 구조화 하는 패턴

마스터-슬레이브 패턴(Master-Slave Pattern): 처리된 결과물을 다시 돌려받는 방식으로 작업을 수행하는 패턴
브로커 패턴(Broker Pattern): 사용자가 원하는 서비스와 특성을 브로커 컴포넌트가 요청에 맞는 컴포넌트와 연결해주는 패턴
피어-투-피어 패턴(Peer-To-Peer Pattern): 피어라 불리는 하나의 컴포넌트가 클라이언트가 될수도, 서버가 될수도 있는 패턴
이벤트-버스 패턴(Event-Bus Pattern): 소스가 특정 채널에 이벤트 메시지를 발행하면, 해당 채널을 구독한 리스너들이 메시지를 받아 이벤트를 처리하는 패턴
블랙보드 패턴(Blackboard Pattern): 모든 컴포넌트들이 공유 데이터 저장소와 블랙보드 컴포넌트에 접근이 가능한 패턴
인터프리터 패턴(Interpreter Pattern): 프로그램 코드의 각 라인을 수행하는 방법을 지정하고, 기호마다 클래스를 갖도록 구성된 패턴

객체지향
각 요소들을 객체로 만든 후, 객체들을 조립하여 소프트웨어 개발하는 기법

객체(Object): 데이터나 이를 처리하기 위한 함수를 묶어놓은 소프트웨어 모듈
클래스(Class): 공통된 속성이나 연산을 갖는 객체의 집합
메시지(Message): 객체들간 상호작용 하는수단으로, 객체의 동작이나 연산을 요구하는 외부의 요구 사항
캡슐화(Encapsulation): 외부의 접근을 막기위해 세부 내용을 은닉
상속(Inheritance): 상위 클래스의 모든 속성과 연산을 하위 클래스가 물려받는것
다형성(Polymorphism): 하나의 메시지에 대해 각각의 객체가 가지고 잇는 고유한 방법으로 응답할 수 있는 능력

연관성(Relationship): 두개이상의 객체들이 서로 상호 참조하는 관계
is member of - 연관화(Association) - 2개 이상의 객체가 연관
is instance of - 분류화(Classification) - 동일한 형의 특성 객체
is part of - 집단화(Aggregation) - 관련 있는 객체 묶어 상위 객체 구성
is a - 일반화(Generalization) : 공통적인 성질로 추상화한 상위 객체 구성
      - 특수화/상세화 (Specialization): 상위 객체를 구체화하여 하위 객체 구성

객체지향분석(OOA: Object Oriented Analysis): 사용자의 요구사항과 관련된 객체, 속성, 연산, 관계 등을 정의하여 모델링하는 작업

럼바우 방법- 객체 모델링(정보 모델링), 동적 모델, 기능 모델
부치 방법 - 미시적 개발, 거시적 개발
jacobson 방법 - 유스케이스 강조
coad yourdon 방법 - er 다이어그램
wirfs-brock 방법 - 고객명세서를 평가해서 설계작업까지 연속적으로 수행

모듈(Module): 모듈화를 통해 분리된 시스템의 각 기능
결합도(Coupling): 모듈간의 상호 의존하는 정도, 낮으면 좋고 높으면 안좋음
결합도 큰 순서
내용 결합도(Content Coupling):한 모듈이 다른 모듈 내부 기능 참조 혹은 수정
공통 결합도(Common Coupling): 공유되는 공통 데이터 영역을 여러 모듈이 사용할 때 겨합도
외부 결합도(External Coupling): 어떤 모듈에서 선언한 데이터를 외부 모듈에서 참조할 때 결합도
제어 결합도(Control Coupling): 어떤 모듈이 다른 모듈의 논리 흐름을 제어하기 위해 제어신호 전달하는 결합도
스탬프 결합도(Stamp Coupling): 모듈간 인터페이스로 자료구조 전달될 때
자료 결합도(Data Coupling): 모듈간 인터페이스가 자료요소로만 구성될 때

응집도
모듈의 내부 요소들이 관련있는 정도, 높으면 좋고, 낮으면 안좋음
응집도 큰 순서
기능적 응집도(Functional Cohesion): 기능이 관련된 것
순차적 응집도(Sequential Cohesion): 모듈 내 하나의 출력 데이터를 그 다음 활동 입력 데이터로 사용
교환적 응집도(Communication Cohesino): 동일한 입력과 출력을 사용하는 
절차적 응집도(Procedure Cohesion): 기능을 순차적으로 수행할 경우 응집도 
시간적 응집도(Temporal Cohesion): 특정 시간에 처리되는 기능
논리적(Logical Cohesion): 유사한 성격, 특정한 형태
우연적 응집도(Coincidental Cohesion): 서로 관련 없는 요소


팬인(Fan-In) : 어떤 모듈을 제어하는 모듈의 수
팬아웃(Fan-Out) : 어떤 모듈에 의해 제어되는 모듈의 수
시스템 복잡도 최적화하려면, 팬인 높게, 팬 아웃 낮게

NS 차트(Nassi Schneiderman Chart) : 논리의 기술에 중점을 두고 도형을 이용해 표현하는 방법

단위 모듈(Unit Module): 여러 동작 중 한 가지 동작을 수행하는 기능을 모듈로 구현한 것

IPC(Inter Process Communication): 모듈 간 통신 방식을 구현하기 위해 사용되는 인터페이스 집합

테스트 케이스(Test Case):소프트웨어가 사용자 요구사항에 정확하게 준수했는데 확인하기위한 테스트 항목에 대한 명세서

공통모듈
여러 프로그램에서 공통적으로 사용할 수 있는 모듈

공통모듈 명세 기법 종류
정확성(Correctness): 해당기능이필요하다는것을 알수 있도록 정확성
명확성(Clarity): 중의적으로 해석되지 않게 명확하게
완전성(Completeness): 시스템 구현에 모든것 기술
일관성(Consistency): 공통 기능간 상호충돌x

재사용(Reuse): 이미 개발된 기능을 새로운 시스템이나 기능 개발에 사용하기 적합하도록 최적화하는 것

디자인 패턴
모듈간의 관계 및 인터페이스를 설계할 때 참조할 수 있는 전형적인 해결 방식

GOF의 디자인 패턴
생성 패턴: 클래스나 객체의 생성과 참조과정을 정의하는 패턴
추상 팩토리(Abstract Factory): 구체적인 클래스에 의존하지 않고, 인터페이스를 통해 서로 연관, 의존하는 객체들의 그룹으로 생성하여 추상적으로 표현
빌더(Builder): 작게 분리된 인스턴스를 건축 하듯이 조합하여 객체를 생성, 생성과정과 표현 방법 분리
팩토리 메소드(Factory Method):  객체 생성을 서브 클래스에서 처리하도록 분리하여 캡슐화한 패턴, 가상 생성자 패턴
프로토타입(Prototype): 원본 객체 복제
싱글톤(SIngleton):하나의 객체 생성

구조 패턴: 클래스나 객체들을 조합하여 더 큰 구조로 만드는 패턴
어댑터(Adapter): 호환성 없는 클래스들의 연결
브릿지(Bridge): 구현부에서 추상층 분리, 서로가 독립적 확장
컴포지트(Composite): 여러 객체를 가진 복합 객체와, 단일 객체를 구분 없이 다루고자 할 때 패턴
데코레이터(Decorator): 객체간 결합통해 능동적으로 기능 확장
퍼싸드(Facade) : 더 상위 인터페이스 구성
플라이웨이트(Flyweight): 인스턴스 가능한 공유
프록시(Proxy) - 접근이 어려운 객체와 연결하려는 객체 사이 인터페이스 역활

행위 패턴: 클래스나 객체들이 서로 상호작용하는 방법이나 책임 분배 방법을 정의하는 패턴
책임 연쇄(Chain of Responsibility): 한 객체가 처리하지 못하면 다음 객체로 넘어가는 형태 패턴
커맨드(Command): 캡슐화하여 재이용하거나 취소할수 있도록
인터프리터(Interpreter): 언어 문법 표현 정의하는 패턴
반복자(Iterator): 접근이 잦은 객체에 대해 동일한 인터페이스를 사용하도록 하는 패턴
중재자(Mediator): 수많은 객체간 복잡한 상호작용 캡슐화
메멘토(memento): 특정 시점으로 돌릴 수 있는
옵서버(Observer): 변화되면 알리는
상태(State): 상태에따라 동작을 다르게 처리
전략(Strategy): 원하는 전략 선택
템플릿 메소드(Template Method): 상위 클래스 골격 정의 하위 클래스 구체화
방문자(Visitor): 각 클래스마다 처리 분리하여 별도 클래스로 구성

통합 개발환경 (IDE)
다양한 툴을 하나의 인터페이스로 통합하여 제공하는 환경

기업에서 사용하는 여러 기능을 통합하여 제공하는 소프트웨어

애플리케이션 테스트
애플리케이션에 잠재되어 있는 결함을찾아내는 절차(확인, 검증)

애플리케이션 테스트의 기본원리
완벽한 테스트 불가능
파레토 법칙(Pareto Principle) -  애플리케이션의 20%에 해당하는 코드에서 전체 결함의 80프로가 발견된다는 법칙
살충제 패러독스(Pesticide Principle) -  동일한 테스트 케이스로 동일한 테스트를 반복하여 더 이상 결함이 발견되지 않는 현상
테스트는 정황 의존 -  정황에 따라 테스트를 다르게 해야함
오류 부재의 궤변(Absence of Errors Fallacy: 오류가 없어도 사용자 요구사항을 만족하지 못하면 해당 소프트웨어 품질은 높다고 말할 수 없는것
테스트와 위험은 반비례 - 테스트를 많이 할 수록 위험을 줄일 수 있음
테스트의 점진적 확대 - 테스트는 작은 부분에서 시작하여 점점 확대하여 진행해야 함
테스트의 별도 팀 수행 - 테스트는 개발자와 관계없는 별도의 팀에서 수행해야 함

정적 테스트
프로그램을 실행하지 않고 소스코드를 대상으로 오류 찾는 테스트
ex) 워크스루(개발자가 모집한 전문가들이 개발자의 작업 내용을 검토), 인스펙션(소개발 단계에서 산출된 결과물 품질 평가하고 이를 개선하기 위한 방법 제시), 코드 검사

동적 테스트
프로그램을 실행하여 오류를 찾는 테스트
ex) 블랙박스 테스트, 화이트박스 테스트

테스트 기반 테스트
명세 기반 테스트 - 사용자 요구사항에 대한 명세를 빠짐없이 테스트 케이스로 만들어 확인
구조 기반 테스트 - 소프트웨어 내부 흐름에 따라 테스트 케이스로 만들어 확인
경험 기반 테스트 - 유사 소프트웨어나 기술에 대한 테스터의 경험을 기반으로 수행하는 테스트

시각에 따른 테스트
검증 테스트 - 개발자 시각에서 제품의 생산 과정을 테스트
확인 테스트 - 사용자의 시각에서 생성된 제품의 결과 테스트

목적에 따른 테스트
회복 테스트 - 여러가지 결함을 준 후 올바르게 복구되는지 확인하는 테스트
안전 테스트 - 불법적인 침입으로부터 시스템을 보호할 수 있는지 확인하는 테스트
강도 테스트 - 과부하시에도 소프트웨어가 정상적으로 실행되는지 확인하는 테스트
성능 테스트 - 실시간 성능이나 전체적인 효율성을 진단하는 테스트
구조 테스트 - 소프트웨어 내부의 논리적인 경로, 소스 코드의 복잡도 등을 평가하는 테스트
회귀 테스트 - 수정된 코드에 결함이 없는지 확인하는 테스트
병행 테스트 - 기본 소프트웨어와 변경된 소프트웨어에 동일한 입력값을 주어 결과를 비교

기초 경로 검사(Base Path Testing): 절차적 설계의 논리적 복잡성을 측정해주는 테스트
제어 구조 검사(Control Structure Testing)
- 조건 검사(Condition Testing): 논리적 조건을 테스트
- 루프 검사(Loop Testing): 루프 구조에 초점을 맞춰 실시하는 테스트
- 데이터 흐름 검사(Data Flow Testing): 변수의 정의나 변수 사용에 위치에 초점을 맞춰 실시하는 테스트

화이트박스 테스트의 검증 기준
문장 검증 기준(Statement Coverage) - 모든 구문이 한 번 이상 수행되도록 설계
분기 검증 기준(Branch Coverage) - 소스 코드의 모든 조건문을 한 번 이상 수행되도록 항목 설계
조건 검증 기준(Condition Coverage) - 소스 코드의 모든 조건문의 참, 거짓을 각 한번 이상 수행되도록 항목 설계
분기/조건 기준(Branch/Condition Coverage) - 소스 코드의 모든 조건문과 각 조건문에 포함된 개별 조건식의 결과가 참, 거짓인 경우 각각을 한 번 이상 수행되도록 항목 설계

블랙박스 테스트(Black Box Test)
각 기능이 완전히 작동하는지 입증하는 테스트

동치 분할 검사(Equivalence Partitioning Testing) - 입력 조건에 타당한 입력과 타당하지 않은 입력 자료의 개수를 동등하게 하여 동일한 결과가 출력되는지 확인하는 기법
경계값 분석(Boundary Value Analysis) - 입력 조건의 경계값을 테스트
원인-효과 그래프 검사(Cause-Effect Graphing Testing) - 입력 데이터간의 관계와 출력에 영향을 미치는 상황을 체계적으로 분석한 다음 효용성이 높은 테스트케이스를 선정하여 검사하는 방법
오류 예측 검사(Error guessing) - 과거 경험이나 확인자의 감각으로 테스트
비교 검사(Comparison Testing) - 여러 버전의 프로그램에 테스트 자료를 제공하여 동일한 결과가 나오는지 테스트\

개발 단계에 따른 애플리케이션 테스트(v모델)
단위 테스트 -> 통합 테스트 -> 시스템 테스트 -> 인수 테스트

단위 테스트
모듈이나 컴포넌트에 초점을 맞춰 테스트

통합 테스트
단위 테스트가 완료된 모듈들을 결합하여 하나의 시스템으로 만드는 과정에서의 테스트

시스템 테스트
개발된 소프트웨어가 해당 컴퓨터 시스템에서 완벽하게 수행되는가 점검하는 테스트

인수 테스트
사용자의 요구사항을 충족하는지에 중점을 두는 테스트
사용자 인수 테스트 - 사용자가 시스템 사용의 적절성 여부를 판단
운영상의 인수 테스트 - 시스템 관리자가 시스팀 인수 시 수행하는 테스트
계약 인수 테스트 - 계약 상의 조건을 준수하엿는지
규정 인수 테스트 -  소프트웨어가 규정에 맞게 개발 되었는지
알파 테스트 - 사용자가 개발자 앞에서 테스트
베타 테스트 - 사용자가 여러 사용자 앞에서 행하는 테스트

통합 테스트(Integration Test)
단위 테스트가 끝난 모듈을 통합하는 과정에서 발생하는 오류 및 결함 테스트 기법

통합 테스트 종류
비점진적 통합 방식 - 단계적으로 통합하는게 아니고 미리 결합되어 잇는 프로그램 전체를 테스트
ex) 빅뱅 통합 테스트
점진적 통합 방식 - 모듈 단위로 단계적으로 통합하면서 테스트하는 방법
ex) 하향식 통합 테스트 ,상향식 통합 테스트, 혼합식 통합 테스트

하향식 통합 테스트(Top Down Integreation Test)
상위 모듈에서 하위 모듈 방향으로 통합하면서 테스트하는 기법 by 깊이 우선 통합법 ,넓이 우선 통합법
테스트 절차
1. 주요 제어 모듈을 작성된 프로그램, 종속 모듈은 스텁으로 대체
2. 깊이 우선 or 넓이 우선 통합 방식에 따라 하위 모듈인 스텁들이 한번에 하나 씩 실제 모듈로 교체
3. 모듈이 통합될 때 마다 테스트
4. 새로운 오류가 발생하지 않았음을 보증하기위해 회귀 테스트 진행

스텁: 일시적으로 필요한 조건만을 가지고 있는 시험용 모듈

상향식 통합 테스트(Bottom Up Integration Test)
하위 모듈에서 상위 모듈 방향으로 통합하면서 테스트하는 기법
절차
1. 하위 모듈들을 클러스터로 결합    (클러스터: 하나의 주요 모듈과 관련된 종속 모듈의 그룹)
2. 상위 모듈에서 데이터 입출력을 확인하기 위해 더미 모듈인 드라이버를 작성
3. 통합된 클러스터 단위로 테스트
4. 테스트가 완료되면 클러스터는 프로그램 구조의 상위로 이동하여 결합하고 드라이버는 실제 모듈로 대체

테스트 드라이버
테스트 대상의 하위 모듈을 호출하고, 파라미터를 전달하고 테스트 결과를 반환

혼합식 통합 테스트(샌드위치 통합 테스트 방법)
하위 수준에서는 상향식 통합, 상위 수준에서는 하향식 통합을 사용하여 최적의 테스트를 지원하는 방법

회귀 테스팅
통합 테스트로 인해 변경된 모듈이나 컴포넌트에 새로운 오류가있는지 확인

테스트 계획 -> 테스트 분석 및 디자인 -> 테스트 케이스 및 시나리오 작성 -> 테스트 수행 -> 테스트 결과 평가 및 리포팅 -> 결함 추적 및 관리

결함 관리 프로세스
에러 발견 -> 에러 등록 -> 에러 분석 -> 결함 확정 -> 결함 할당 -> 결함 조치 -> 결함 조치 검토 및 승인

테스트 케이스
사용자 요구사항을 준수햇는지 확인하기 위한 테스트 항목에 대한 명세서

테스트 시나리오
여러개의 테스트 케이스가 잇을때 이를 적용하는 순서

테스트 오라클
테스트 결과가 맞는지 확인하는 도구
특징
제한된 검증 - 모든 테스트 케이스에 적용할 수 없음
수학적 기법 - 테스트 오라클의 값을 수학적 기법을 이용하여 구할 수 있음
자동화 가능 - 테스트 대상 프로그램 실행, 결과 비교, 커비리지 측정 등을 자동화 할 수 있음

테스트 오라클의 종류
참 오라클 - 모든 테스트 케이스의 입력값에 대해 기대하는 결과를 제공하는 오라클
샘플링 오라클 - 특정한 몇몇 케이스의 입력 값에 대해서만 기대하는 결과를 제공하는 오라클
추정(Heuristic) 오라클 - 특정 테스트 케이스의 입력 값에 대해 기대하는 결과를 제공하고, 나머지 입력 값들은 추정으로 제공하는 오라클
일관성(Consistent) 검사 오라클 - 애플리케이션 변경이 있을떄 테스트 케이스 수행 전과 후의 결과 값이 동일한지 확인하는 오라클

테스트 자동화
사람이 반복적으로 수행하던 테스트 절차를 스크립트 형태로 구현하는 자동화 도구를 적용함으로써 쉽고 효율적으로 테스트를 수행할 수 있도록 한 것

종류
정적 분석 도구(Static Analysis Tools)
프로그램을 실행하지 않고 분석하는 도구

테스트 실행 도구(Test Execution tools)
스크립트 언어를 사용하여 테스트를 실행하는 도구(데이터 주도 접근 방식, 키워드 주도 접근 방식)
성능 테스트 도구(Performance test Tools)
가상의 사용자를 만들어 성능 목표 달성을 확인하는 도구
테스트 통제 도구(Test Control Tools)
테스트 계획 및 관리, 테스트 수행, 결함 관리 등 수행하는 도구
테스트 하네스 도구(Test Harness Tools)
테스트가 실행될 환경을 시뮬레이션 하여 컴포넌트 및 모듈이 정상적으로 수행되는지 테스트되도록 하는 도구

테스트 하네스의 구성요소
테스트 드라이버 - 테스터 대상의 하위 모듈을 호출하고, 파라미터를 전달하고, 테스트를 수행한 결과값을 반환하는 도구
테스트 스텁 - 일시적으로 필요한 조건을 가지고 있는 테스트용 모듈
테스트 슈트 - 테스트 케이스의 집합
테스트 케이스 - 사용자 요구사항 준수했는지 확인하기위한 테스트 항목의 명세서
테스트 스크립트 -  자동화된 테스트 절차에 대한 명세서
목 오브젝트 - 사전에 행위를 입력해두면, 예정된 행위를 실행하는 객체

테스트 수행 단계별 테스트 자동화 도구
테스트 계획 - 요구사항 관리
테스트 분석/설계 - 테스트 케이스 생성
테스트 수행 - 테스트 자동화, 정적 분석, 동적 분석, 성능 테스트, 모니터링
테스트 관리 - 커버리지 분석, 형상 관리, 결함 추적/관리

애플리케이션 성능

데이터베이스 설계 순서
요구 조건 분석: 데이터베이스 사용할 사람들로부터 필요한 용도 파악 - 요구 조건 명세서 작성
개념적 설계: 현실 세계에 대한 인식을 추상적 개념으로 표현 - 개념스키마, 트랜잭션 모델링, ER 모델
논리적 설계: 현실 세계에 발생하는 자료를 특정 DBMS가 지원하는 논리적 자료구조로 변환 - 목표 DBMS에 맞는 논리스키마 설계, 트랜잭션 인터페이스 설계
물리적 설계 - 목표 DBMS에 맞는 물리적 구조의 데이터로 변환
구현: 논리적 설계와 물리적 설계에서 도출한 데이터베이스 스키마를 파일로 생성하는 과정 - DDL로 데이터베이스 생성

데이터 모델 구성 요소: 개체, 속성, 관계
데이터 모델 종류: 개념적 데이터 모델, 논리적 데이터 모델, 물리적 데이터 모델
데이터 모델 표시할 요소: 구조, 연산, 제약조건

개념적 데이터 모델

논리적 데이터 모델

데이터 모델 표시할 요소
구조(Structure): 논리적으로 표현된 개체들 간의 관계로 데이터 구조 및 정적 성질을 표현
연산(Operation): 실제 데이터를 처리하는 작업 명세로서 데이터베이스를 조작하는 기본 도구
졔약조건(Constraint): 실제 데이터의 논리적인 제약조건

개체(Entity)
데이터베이스에 표현하려는 거으로 개념이나 정보 단위 같은 현실 세계의 대상체

속성(속성의 수를 차수(degree))
데이터베이스를 구성하는 가장 작은 논리적 단위

속성의 특성에 따른 분류
기본 속성(Basic Attribute)
설계 속성(Designed Attribute)
파생 속성(Derived Attribute)

속성의 개체 구성 방식에 따른 분류
기본키 속성 - 개체를 유일하게 식별할 수 있는 속성
외래 키 속성 - 다른 개체와의 관계에서 포함된 속성
일반 속성 - 개체에 포함되어 있고, 기본키, 외래키에 포함되지 않은 속성

관계의 종류
종속 관계(Dependent Relationship) 주, 종 관계 표현
중복 관계(Redundant Relatoinship) 두 개체 사이 2번 이상 종속 관계 발생
재귀 관계(Recursive Relationship) 개체가 자기 자신과 관계를 갖는 것
배타 관계(Exclusive Relationship): 구분자를 기준으로 분할하는 관계

식별자
하나의 개체 내에서 각각의 인스턴스를 유일하게 구분할 수 있는 구분자

식별자의 분류
대표성 여부
- 주 식별자: 개체를 대표하는 유일한 식별자
- 보조 식별자: 주 식별자를 대신하여 개체를 식별할 수 있는 속성
스스로 생성 여부
-내부 식별자: 개체 내에서 스스로 만들어지는 식별자
-외부 식별자: 다른 객체와 관계에 의해 외부 식별자를 가져와 사용하는 식별자
단일 속성 여부
-단일 식별자: 주 식별자가 한가지 속성
-복합 식별자: 주 식별자가 두 개 이상의 속성
대체 여부
- 원조 식별자: 업무에 의해 만들어지는 가공되지 않은 원래의 식별자
- 가공 식별자: 주 식별자 속성이 두개이상인 경우 속성들을 하나의 속성으로 묶어 사용하는 식별자

후보 식별자
개체에서 각 인스턴스를 유일하게 식별할 수 있는 속성 또는 속성집합

관계 해석(Relational Calculus): 관계 데이터의 연산을 표현하는 방법

이상(Anomaly)
테이블에서 데이터들의 중복이 발생하고, 이 중복으로 인해 문제가 발생하는 현상

삽입 이상(Insertion Anomaly)
테이블에 데이터를 삽입할 때 의도와는 관계없이 원하지 않는 값들로 인해 데이터를 삽입할 수 없는 현상

삭제 이상(Deletion Anomaly)
테이블에서 한 튜플을 삭제할 때, 의도와는 상관없는 값들도 삭제되는 현상

갱신 이상(Update Anomaly)
테이블에 튜플의 속성 값을 갱신할 때 일부 튜플만 수정되어 정보에 불일치성이 생기는 현상

함수적 종속(Functional Dependency)
속성 X의 값 각각에 대해 항상 Y의 값 하나만 연관되어 있을 때를 Y를 X에 함수정 종속이라 한다. (X -> Y)
X를 결정자, Y를 종속자

정규화(Normalization)-
테이블의 속성들이 상호 종속적인 관계를 갖는다는 특성을 이용하여 테이블을 무손실 분해하는 과정

제1정규형
테이블의 모든 속성의 도메인이 원자값만으로 되어 있는 정규형

제2정규형
제1정규형을 만족하고 기본키가 아닌 모든 속성이 기본키에 대해 완전 함수적 종속을 만족

제3정규형
제2정규형을 만족하고 기본키가 아닌 모든 속성이 기본키에 대해 이행적 함수적 종속을 만족하지 않는 정규형

BCNF
제3정규형을 만족하는 테이블에서 모든 결정자는 후보키인 정규형

제4정규형
테이블에 다중값종속(다치 종속) A ->->B 이 존재할 경우 모든 속성이 A에 함수적 종속 관계를 만족하는 정규형

제5정규형
테이블의 모든 조인 종속이 후보키를 통해서만 성립되는 정규형

도메인 원자, 부분 함수 종속 제거, 이행적 함수 종속 제거, 결정키 후보키 아닌것 제거, 다치 종속제거, 조인 종속성 이용

반 정규화(Denormalization)
정규화된 데이터 모델을 의도적으로 통합, 중복, 분리하여 정규화 원칙을 위배하는 행위

뷰(view), (CREATE 정의, 제거할때 DROP)

장점: 논리적 데이터 독립성을 제공, 자동보안
단점: 독립적인 인덱스 가질수 없음, 뷰의 정의를 변경할 수 없음

조회속도 향상, 입력, 수정, 삭제에 대한 작업 성능 저하

파티션(Partition)
대용량의 테이블이나 인덱스를 작은 논리적 단위인 파티션으로 나누는 것
장점:쿼리 성능 향상
단점:용량이 작은 테이블에 하면 오히려 성능 저하, 테이블간 조인에 대한 비용 증가, 세심 관리 요구

파티션 종류

조합 분할(Composite Partitioning) - 범위 분할 적용한뒤 해시 분할 적용하여 분할

분산 데이터베이스 설계
논리적으로 하나의 시스템에 속하지만 물리적으로 네트워크를 통해 연결된 여러개의 사이트에 분산된 데이터베이스를

분산 데이터베이스의 목표
위치 투명성(Location Transparency): 데이터베이스의 실제 위치를 알필요 없음
중복 투명성(Replication Transparency): 동일 데이터가 여러 곳에 중복되어 잇어도 마치 하나의 데이터만 잇는 것 처럼 사용
병행 투명성(Concurrcny Transparency): 분산 데이터베이스와 관련된 다수의 트랜잭션이 동시에 실현되더라도 그 트랜잭션의 결과는 영향을 받지 않음
장애 투명성(Failure Transparency): 장애애도 불구하고 트랜잭션을 정확하게 처리

분산 설계 방법
테이블 위치 분산: 데이터베이스의 테이블을 각기 다른 서버에 배피
분할(Fragmentation): 테이블의 데이터를 분할하여 분산(수평 분할, 수직 분할)
할당(Allocation): 동일한 분할을 여러개의 서버에 생성하는 분산 방법

DAS 빠른 처리, NAS 파일 공유 장점을 혼합한 방식으로, 서버와 저장장치를 연결하는 전용 네트워크를 별도로 구성하는 방법

배열(Array)
크기와 형이 동일한 자료들이 순서대로 나열된 자료의 집합

연속 리스트(Contiguous List)
연속되는 기억장소에 저장되는 자료구조

연결 리스트(Linked List)
자료를 임의의 공간에 저장시키되 노드의 포인터 부분을 이용하여 서로 연결시킨 자료구조

스택은 리스트의 한쪽 끝으로만 자료 삽입, 삭제 작업이 이루어지는 자료구조

큐는 리스트의 한쪽에서는 삽입 작업, 다른 한쪽에서 삭제 작업이루어지는 자료구조

그래프는 정점과 간선의 두 집합으로 이루어지는 자료 구조

방향 그래프 최대 간선 수: n(n-1)
무방향 그래프 최대 간선 수: n(n-1) / 2

트리
정점과 선분을 이용하여 사이클을 이루지 않도록 구성하는 그래프

디그리:각 노드에서 뻗어나온 가지의 수
단말 노드 = 리프 노드: 자식이 하나도 없는 노드
비단말 노드 = 자식이 하나라도 잇는 노드

이진트리
차수가 2이하인 노드들로 구성된 트리
이진트리 레벨 i에서 최대 노드의수 2^(n-1)
n0 = n2 + 1

트리 운행법(Tree Traversal)
preorder -  root -> left -> right
inorder - left -> root -> right
postorder - left -> right -> root

삽입 정렬(Insertion Sort)
이미 순서화된 파일에 새로운 레코드 하나를 순서에 맞게 삽입시켜 정렬 방식

선택 정렬(Selection Sort)
n개의 레코드 중 최소값을 찾아 첫번쨰 레코드에 놓고, 나머지 n-1개 중 다시 최소값을 찾아 두번쨰 레코드에 놓는 방식을 반복하여 정렬하는 방식

버블 정렬(Bubble Sort)
인접한 두개의 레코드 키 값을 비교하여 크기에 따라 레코드 위치를 서로 교환하는 정렬 방식

쉘 정렬(Shell Sort)
어떤 매개변수의 값으로 서브파일을 구성하고, 각 서브 파일을 Insertion 정렬 방식으로 순서 배열하는 정렬 방식

퀵 정렬(Quick Sort)
키를 기준으로 작은 값은 왼쪽 큰값은 오른쪽 서브파일에 분해시키는 정렬 방식

힙정렬(Heap Sort)
완전 이진 트리를 이용한 정렬 방식

2 Way 합병 정렬(Merge Sort)
이미 정렬되어 잇는 두개의 파일을 한개의 파일로 합병하는 정렬방식

기수 정렬(Radix Sort = Bucket Sort)
Queue를 이용하여 자릿수 별로 정렬하는 방식

기억장치 관리 전략
보조기억장치의 프로그램이나 데이터를 주기억장치에 적재시키는 시기, 위치 등을 지정하여 한정된 주기억장치 공간을 효율적으로 사용하기 위한 것
종류
반입(Fetch) 전략
프로그램이나 데이터를 언제 주기억장치에 적재할것인지 결정
요구 반입(Demand Fetch):  실행중인 프로그램이 특정 프로그램이나 데이터를 참조를 요구할 떄 적재
예상 반입(Anticipatory Fetch): 실행중인 프로그램에 의해 사용될 프로그램이나 데이터를 미리 예상해서 적재

배치(placement) 전략
프로그램이나 데이터를 주기억장치에 어디에 위치 시킬 것인지 결정
최초 적합(First Fit): 프로그램이나 데이터가 들어갈 빈 영역중 첫번 째
최적 적합(Best Fit): 프로그램이나 데이터가 들어갈 빈 영역 중 단편화를 가장 적게 남기는 곳에 배치
최악 적합(Worst Fit): 프로그램이나 데이터가 들어갈 빈 영영 죽 단편화가 가장 많이 남기는 곳에 배치

교체(Replacement) 전략
이미 사용하고 있는 영역중 어느 영역을 교체하여 사용할 것인지 결정하는 전략

주기억장치 할당
프로그램이나 데이터를 실행시키기 위해 주기억장치에 어떻게 할당할 것인가

주기억장치 할당기법 분류
연속 할당 기법: 프로그램을 주기억장치에 연속하여 할당하는 기법(단일 분할 할당기법, 다중 분할 할당 기법)
분산 할당 기법: 프로그램을 특정 크기로 나누어 분산하여 저장하는 할당하는 기법(종류: 페이징 기법, 세그먼테이션 기법)

단일 분할 할당 기법
한순간에는 오직 한명의 사용자만이 주기억 장치의 사용자 영역을 사용
단일 분할 할당 기법 종류: 오버레이(Overlay): 주기억 장치 보다 큰 프로그램을 실행하기 위한 기법, 스와핑(Swapping): 하나의 프로그램 전체를 주기억 장치에할당하여 사용하다 필요에 따라 다른 프로그램과 교체하는 기법

경계 레지스터: 사용자 영역에 있는 사용자가 운영체제 영역에 접근하지 못하도록 하는 기법

다중 분할 할당 기법
종류
고정 분할 할당 기법(정적 할당 기법): 미리 고정된 크기 분할하여 할당
가변 분할 할당 기법(동적 할당 기법) : 미리 분할하지 않고 필요에 따라 분할

가상기억장치(Virtual Memory): 보조기억장치의 일부를 주기억장치처럼 사용하는 것

페이징 기법(Paging)
프로그램과 주기억장치의 영역을 동일하게 나눈 후 나눠진 프로그램을 주기억 장치에 적재하여 실행하는 기법
외부 단편화는 발생하지 않지만, 내부 단편화 발생

세그멘테이션 기법(Segmentation)
프로그램을 다양한 크기의 논리 단위로 나눈 후 주 기억장치에 적재시켜 실행하는 기법
내부 단편화는 바생하지 않지만 외부단편화는 발생

페이지 교체 알고리즘
페이지 부재 발생하면 가상기억장치에서 필요한 페이지를 적재해야되는데, 이 때 주기억장치의 모든 페이지가 사용중이라면 어떤 페이지 프레임을 선택하여 교체할지 결정하는 기법

페이지 부재: CPU가 액세스한 페이지가 주기억장치에 없는 경우

최적교체(OPTimal replacement)
앞으로 가장 오랫동안 사용하지 않을 페이지를 교체

FIFO(First in First Out)
가장 먼저 들어와서 오래 있떤 페이지를 교체하는 기법

LRU(Least Recently Used)
최근 가장 오랫동안 사용하지 않은 페이지를 교체하는 기법

LFU (Least Frequently Used)
사용빈도가 가장 적은 페이지 교체하는 기법

NUR(Not Used Recently)
최근에 사용하지 않은 페이지를 교체

SCR(Second Chance Replacement) 2차 기회 교체
가장 오랫동안 주기억장치에 있던 페이지 중 자주 사용하는 페이지의 교체를 방지하기 위한 기법, FIFO 단점 보완

페이지 크기
페이지 크기가 작을 경우
페이지 단편화 감소, 한개의 페이지를 주기억장치로 이동시키는 시간이 줄어듦, 디스크 접근횟수가 많아져서 전체적인 입/출력 시간은 늘어남

페이지 크기가 클 경우
페이지 단편화 증가, 한개의 페이지 주기억장치로 이동시키는 시간이 많이듬, 디스크 접근횟수가 줄어들어 전체적인 입/출력 효율성 증가

Locality
프로세스가 실행되는동안 주기억장치를 참조할때 일부 페이지만 집중적으로 참조하는 성질

시간 구역성(Temporal Locality) - 프로세스가 실행되면서 하나의 페이지를 일정시간동안 집중적으로 액세스하는현상
공간 구역성(Spartial Locality) - 프로세스가 실행시 특정 위치의 페이지를 집중적으로 액세스하는 현상

워킹 셋(Working Set)
프로세스가 일정시간 동안 자주 참조되는 페이지들의 집합

스레싱(Thrashing)
프로세스 처리 시간보다 페이지의 교체 시간이 더 많이 소요되는 현상

프로세스
실행중인 프로그램

프로세스가 시스템 내에 존재하는 동안 프로세스 상태가 변하는 것

시스템의 여러 자원을 할당받아 실행하는 프로그램의 단위

Dispatch
준비 상태로 대기하고 있던 프로세스중 하나가 프로세서를 할당받아 실행 상태로 전이되는 과정
Wake Up
대기 상태에서 준비 상태로 전이 되는 과정
Spolling
데이터를 한번에 디스크에 저장하는 과정
교통량 제어기(Traffic Controller)
프로세스 상태에 대한 조사와 통보 담당

스케줄링
시스템의 여러 자원을 해당 프로세스에게 할당해주는 과정

스케줄링 종류
장기 스케줄링
어떤 프로세스가 시스템의 자원을 차지할 수 있을 것인가 결정하여 준비 상태 큐로 보내는 작업
중기 스케줄링
어떤 프로세스가 CPU 할당받을지 정하는 작업
단기 스케줄링
프로세스가 실행하기 위해 CPU를 할당받는 시기와 특정 프로세스를 지정하는 작업

스케줄링 목적
공정성

균형 있는 자원의 사용
무한 연기 회피

비선점 스케줄링 (Non-Preemptive)
이미 할당된 CPU를 다른 프로세스가 강제로 빼앗아 사용할수 없는 스케줄링 기법
종류: FCFS,  SJF, 우선순위, HRN, 기한부

선점 스케줄링(Preemptive)
우선순위가 높은 다른 프로세스가 CPU를 강제로 빼앗아 사용할 수 있는 스케줄링 기법
종류: ROUND ROBIN, SRT, 선점 우선 순위, 다단계 큐, 다단계 피드백 큐

FCFS(First Come First Service)
준비 상태 큐에 도착한 순서에 따라 CPU를 할당하는 기법

대기시간: 프로세스가 대기한 시간
반환 시간: 프로세스 대기 시간 + 실행 시간의 합

SJF(Shortest Job First) 단기 작업 우선
실행 시간이 가장 짧은 프로세스 먼저 CPU를 할당하는 기법

HRN(Highest Response-ratio Next)
대기시간과 실행 시간을 이용하는 기법, 우선순위 계산하여 높은것 부터 처리
우선순위 계산식 = 대기시간 + 실행시간 / 서비스 시간

인터넷(Internet)
전 세계 수많은 컴퓨터와 네트워크들이 연결된 광범위한 컴퓨터 통신망

IP 주소(Internet Protocol Address)
인터넷에 연결된 컴퓨터 자원 구분하기 위한 고유한 주소(8비트씩 4부분, 총 32비트)

   길이에에 따라 A클래스에서 E클래스까지 총 5단계
A Class
국가나 대형 통신망 ( 2^24) (0~ 127)
B class
중대형 통신망(2^16) (128 ~ 191)
c class
소형 통신망(2^8) (192 ~ 223)
D 클래스
멀티캐스트 용으로 사용(224~ 239)
E 클래스
실험적 주소

서브네팅(Subnetting)
네트워크 주소를 다시 여러개의 작은 네트워크로 나누어 사용하는 것
네트워크 주소와 호스트 주소를 구분하기 위해 사용하는 비트를 서브넷 마스크

IPv6(Internet Protocl Version 6)
Ipv4 주소 부족 문제를 해결하기 위해 개발
128비트(16비트씩 8부분)
IPv4에 비해 전송속도가 빠르다
인증성, 기밀성, 데이터 무결성
확장성, 융통성, 연동성

OSI 참조 모델
다른 시스템과 원활한 통신을 위해 ISO(국제표준기구)에서 제안한 통신 규약

네트워크 인터페이스 카드(NIC, Network Interface Card)
컴퓨터와 컴퓨터, 컴퓨터와 네트워크 사이를 연결하는 장치
이더넷 카드(LAN 카드), 네트워크 어댑터

허브(Hub)
회선을 통합하여 관리
더미 허브: 네트워크에 흐르는 모든 데이터를 단순히 연결
스위칭 허브: 네트워크상에 흐르는 데이터의 유무 및 흐름을 제어하여 각각의 노드가 허브의 최대 대역폭을 사용할 수 있는 지능형 허브

리피터(Reapter)
수신한 신호를 재생시키거나 출력 높여서 전송하는 장치

브리지(Bridge)
LAN과 LAN을 연결하거나 LAN안에서 컴퓨터그룹 연결
- 브리지 이용한 서브넷 구성시 전송 가능한 회선 수 브리지 n -> n(n-1)/2

스위치(Switch)
LAN과 LAN을 연겨랗여 더 큰 LAN을 만드는 장치

라우터(Router)
데이터 전송하는데 최적 경로 선택

게이트웨이(Gateway)

프로토콜(Protocol)
데이터 교환을 원활하게 수행할 수 있도록 표준화 시켜놓은 통신 규약
기본요소

네트워크
두 대 이상의 컴퓨터를 연결하여 자원을 공유하는 것

성형
중앙 컴퓨터가 잇고, 이를 중심으로 컴퓨터들이 연결

링형
컴퓨터와 서로 이웃하는것 연결, 투포인트 방식

버스형
한 회선에 여러개 단말장치 연결

계층형
일정 지역까지 통신회선이 반복

망형
모든 지점 컴퓨터를 서로 연결
근거리 통신(LAN) - 빠름, 낮은 에러 발생- 주로 버스형, 링형

원거리 통신(WAN) - 느림, 높은 에러발생

NAT(Network Address Translation, 네트워크 주소 변환):  한개의 정식 IP에 대량의 가상 사설 IP 할당


스위치는 LAN과 LAN을 연결하여 훨씬 더 큰 LAN을 만드는 장치

L2 스위치 2계층 - MAC 주소 기반(동일 네트워크간 연결만)
L3 스위치 3계층 - L2에 라우터 기능 추가, IP주소기반 패킷 전송, 서로 다른네트워크 연결가능
L4 스위치 4계층 - 로드밸런싱 달링 L3
L7 스위치 7 계층 - 패킷 내용까지 참조해 세밀하게 로드밸런싱

병행 제어
동시에 실행되는 트랜잭션 간의 상호작용을 제어

병행 제어 기법 종류
로킹(Locking) - 어떤 로킹 단위 액세스하기전에 Lock
타임 스탬프 순서(Time Stamp Ordering) - 부여된 시간에 따라 트랜잭션 작업 수행
최적 병행수행(낙관적 기법) - 읽기 전용 트랜잭션은 병행 제어 기법 사용x
다중 버전 기법 - 버전을 부여하여 관리

로킹 단위(Locking Granularity)
한꺼번에 로킹 할 수 잇는 객체 크기
로킹단위크면 관리하기 쉽지만, 병행성 수준 낮아짐
로킹단위작으면 관리하기 어려워 오버헤드 증가하지만, 병행성 수준 높아짐

교착 상태(Dead lock)
서로 다른프로세스가 점유하고 있는 자원을 기다리며 무한정 기다리는 상태

교착 상태 필요 충분조건
상호배제(Mutual Exclsion) - 한번에 한개의 프로세스만 공유 자원 사용
점유와 대기(Hold and Wait) - 최소 하나 자원 점유하고 있으면서 다른프로세스 대기
비선점 (Non-preemptive)- 강제로 빼았을 수 없음
환형 대기(Circurlar Wait) - 두개이상 프로세스가 자원을 기다리는데, 관계에 사이클이 존재
